**M4 - Project Completion (End of Week 10)**

- Application deployed and demo-ready
- **Gate Criteria**: All Chapter 3 objectives met with documented evidence

## Task Dependencies and Critical Path

The following dependency chain represents the **critical path** that determines minimum project duration. Delays in any critical path task will directly impact the final delivery date:

**Critical Path Sequence:**
1. Requirements Gathering (W1-2) → 
2. System Design (W2-3) → 
3. Authentication Implementation (W4-5) → 
4. Order Placement Development (W6) → 
5. Payment Integration (W7) → 
6. Complete Feature Set (W8) → 
7. User Testing (W9-10) → 
8. Project Delivery (W10)

**Key Dependencies:**
- **Authentication ← Development Environment Setup**: Cannot implement login until Firebase configured
- **Order Placement ← Authentication**: Users must be authenticated to place orders
- **Payment Integration ← Order Placement**: Orders must exist before payment processing
- **Loyalty Points ← Payment Integration**: Points earned when payments complete
- **User Testing ← All Features Complete**: Meaningful testing requires complete system
- **Final Documentation ← Testing Results**: Chapter 7 depends on test outcomes

## Time Allocation by Phase

**Total Project Effort**: 150-200 hours over 10 weeks (15-20 hours/week)

| Phase | Duration | Effort Hours | % of Total | Key Activities |
|-------|----------|--------------|------------|----------------|
| **Phase 1: Research & Design** | Weeks 1-3 | 45-60 hours | 30% | Literature review, requirements, system design, wireframes |
| **Phase 2: Development** | Weeks 4-8 | 75-100 hours | 50% | Coding all features, Firebase setup, M-Pesa integration |
| **Phase 3: Testing & Delivery** | Weeks 9-10 | 30-40 hours | 20% | Testing, user feedback, bug fixes, final documentation |

**Weekly Breakdown:**
- **Weeks 1-3**: 15 hours/week (research, design, documentation)
- **Weeks 4-8**: 18-20 hours/week (intensive development period)
- **Weeks 9-10**: 15 hours/week (testing, refinement, report writing)

## Risk-Adjusted Schedule

**Buffer Time**: Built-in contingency within each phase
- Phase 1: 0.5 weeks built into Week 3 (early environment setup completion creates buffer)
- Phase 2: M-Pesa integration spans 2 weeks (W7-8) allowing extra time for complex API
- Phase 3: Parallel bug fixing and documentation (W9-10) provides flexibility

**Schedule Acceleration Options** (if ahead by Week 6):
- Implement optional features (ratings/reviews, favorites)
- Enhanced UI polish with animations
- iOS version development (Flutter enables quick compilation)
- Advanced analytics dashboard features

**Schedule Compression Strategies** (if behind by Week 7):
- Reduce admin dashboard to essential features only (view orders, update status)
- Simplify loyalty points (fixed percentage discount vs tiered rewards)
- Skip optional payment methods (focus M-Pesa only, defer card payments)
- Reduce user testing sample (10 users minimum instead of 15)
