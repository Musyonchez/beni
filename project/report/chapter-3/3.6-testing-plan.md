# 3.6 Testing Plan for the System

The FarmLink prototype was subjected to three levels of testing to verify that the system functioned correctly, securely, and in a manner that met the expectations of its intended users.

## Unit Testing
Unit testing was conducted on individual backend API endpoints and frontend components in isolation. Each API route was tested using Postman to verify that it returned the correct HTTP status codes, response structures, and error messages under both valid and invalid inputs. Frontend components were tested individually to confirm correct rendering and state management behaviour.

## Integration Testing
Integration testing verified that the system's modules worked correctly when connected together. The full user journey was tested end-to-end for each primary use case:
- Farmer registers → creates listing → receives order → updates order status
- Buyer registers → searches produce by location → adds to cart → initiates M-Pesa payment → tracks order → leaves review
- Admin logs in → views reports → manages users → removes a listing

Integration tests confirmed that data flowed correctly between the Next.js frontend, the Node.js backend, and the MongoDB database, and that external service integrations (M-Pesa Daraja sandbox, Twilio SMS) triggered as expected.

## User Acceptance Testing (UAT)
User Acceptance Testing was conducted with a small group of real users — three farmers and two buyers from the study's respondent pool — who were invited to use the FarmLink prototype in their web browser under the researcher's observation. Participants were asked to complete specific tasks (list a product, search for nearby produce, place an order) without assistance, while the researcher noted any points of confusion, errors, or usability concerns. Feedback from UAT informed final UI adjustments made before the prototype was documented in Chapter Seven.
