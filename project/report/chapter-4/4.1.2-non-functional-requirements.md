# 4.1.2 Non-Functional Requirements

Non-functional requirements define the quality attributes and operational constraints of the FarmLink system. These requirements govern how the system performs its functions rather than what those functions are.

| Req. ID | Category | Requirement |
|---|---|---|
| NFR01 | Security | All user passwords shall be hashed using bcrypt before storage. Plaintext passwords shall never be stored or transmitted. |
| NFR02 | Security | All protected API routes shall require a valid JWT token. Tokens shall expire after 7 days and must be re-issued upon expiry. |
| NFR03 | Security | All data transmitted between the mobile client and the server shall be encrypted via HTTPS. |
| NFR04 | Usability | A first-time user shall be able to complete a produce search and place an order within 5 minutes of opening the application without external assistance. |
| NFR05 | Usability | All error messages displayed to the user shall be written in plain, non-technical language and shall suggest a corrective action. |
| NFR06 | Performance | All API responses shall be returned within 2 seconds under normal network conditions (3G or above). |
| NFR07 | Performance | The produce listing feed shall load a maximum of 10 items per page to minimise data consumption on low-bandwidth connections. |
| NFR08 | Availability | The system shall be accessible 24 hours a day, 7 days a week, including for produce browsing and order placement outside of business hours. |
| NFR09 | Compatibility | The mobile application shall be compatible with Android 8.0 (API level 26) and above, and iOS 13 and above. |
| NFR10 | Scalability | The MongoDB database schema shall be designed to support horizontal scaling as the number of users, listings, and orders grows beyond the prototype stage. |
| NFR11 | Maintainability | All backend API routes shall be modularised by domain (auth, products, orders, payments, reviews, notifications) to facilitate independent maintenance and extension. |
